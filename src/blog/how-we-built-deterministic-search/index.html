Building Kobitab

How we built deterministic search

The model gets a cited context bundle that Kobitab has already searched, ranked, and filtered on device.

July 13, 2026 6 min read

We recently rebuilt how chat retrieval works in Kobitab.

Kobitab now gathers the evidence before the model receives a request. It searches approved sources, follows explicit relationships, ranks the results, applies privacy rules, and compiles a small context bundle. The model uses that bundle to write the answer.

Where we started

Our earlier chat flow looked roughly like this:

User message
  -> intent classifier
  -> search over query variants
  -> first useful result set
  -> provider prompt
  -> model answer

It worked for simple lookups. As we added calendar events, email, hosted documents, transcripts, and entity relationships, it became harder to reason about. Search could settle on the first useful result set. Provider prompts accumulated history and metadata. When an answer looked wrong, tracing why a source appeared took too much work.

The new retrieval flow

User message
  -> intent and entity extraction
  -> source search
  -> entity resolution
  -> bounded graph expansion
  -> ranking and deduplication
  -> privacy and budget filtering
  -> ContextBundle
  -> model answer

Given the same user, profile, data snapshot, query, and policy, Kobitab should produce the same context bundle. That bundle is the evidence available to the model.

Understanding the request

Say the user asks:

What do we have for the meeting with Northstar?

Kobitab converts the question into a structured request:

{
  task: "find_context",
  entities: [
    {
      kind: "org",
      text: "Northstar",
      aliases: ["Northstar Labs", "North Star"]
    }
  ],
  concepts: ["meeting"],
  sourceHints: ["calendar", "documents", "email", "video"],
  timeHint: null
}

Rules, local entity data, known aliases, and profile configuration handle this step. The result gives the retrieval layer a stable search plan.

Searching approved sources

Kobitab searches each relevant, profile-scoped source:

  • Calendar events
  • Local and hosted documents
  • Email
  • Video and transcript data
  • Bookmarks and indexed pages

Every adapter returns the same core details: source ID, profile ID, title, timestamp, matched terms, access scope, and stable scoring inputs. Search runs across the approved sources before ranking begins.

Resolving relationships

A direct match is usually the start. A Northstar calendar event may have attendees, attachments, related documents, or a recording. Those items can lead to more evidence through explicit links.

Northstar
  -> Northstar planning meeting
  -> attendees: Jane, Mark
  -> attached document: Q3 launch checklist
  -> linked transcript: Product walkthrough

The graph records facts such as mentions, created_by, attended_by, attached_to, and links_to. Every edge comes from indexed data or a configured alias.

Keeping expansion bounded

If Jane attended the meeting, pulling in everything Jane created would add a lot of weak context. We use hard limits for hops, sources, access, and result counts.

  • Start with exact Northstar matches.
  • Include directly connected events, documents, emails, and videos.
  • Include people attached to those results.
  • Follow another link when its evidence overlaps with the original topic.
  • Stop when a configured cap is reached.

The result contains useful surrounding context while staying predictable.

Ranking the evidence

Each candidate receives a score from explainable features:

  • Exact organization and title matches
  • Recency and source priority
  • Event proximity
  • Author or attendee relationships
  • Explicit attachments and links
  • Query term overlap
  • Access confidence

A score record looks like this:

{
  itemId: "doc_123",
  score: 0.87,
  reasons: [
    "exact_match:northstar",
    "linked_from_calendar_event",
    "attendee_author_overlap",
    "term_overlap:launch"
  ]
}

The reasons help us debug retrieval and explain why an item entered the answer context.

Compiling a safe context bundle

The ranked results are reduced to a compact contract:

type ContextBundle = {
  requestId: string;
  profileId: string;
  query: string;
  policy: ContextPolicy;
  entities: ResolvedEntity[];
  sources: ContextSource[];
  evidence: ContextEvidence[];
  relationships: ContextRelationship[];
  omissions: ContextOmission[];
};

The bundle contains selected excerpts, citations, relationships, and omission details. Full documents, raw provider payloads, hidden metadata, and unbounded database rows stay outside the prompt.

Omission data records what happened during compilation:

{
  candidatesSeen: 184,
  omittedByBudget: 161,
  omittedByAccess: 3
}

That gives us a visible record of privacy filtering and context limits.

Handing context to the model

The provider receives the current question, the context bundle, a small amount of conversation state when needed, and clear answer rules.

The model can summarize findings, compare evidence, explain gaps, and draft follow-up actions. Citations stay connected to the original documents, events, and excerpts. Incomplete evidence produces an explicit gap in the answer.

What we gained

  • Repeatable retrieval for the same data and policy
  • Clear control over what leaves the device
  • Explainable ranking decisions
  • Bounded relationship expansion
  • Smaller provider prompts
  • Better citations and easier debugging

The deterministic layer owns search, relationships, ranking, access control, and context construction. The model handles reasoning and writing over supplied evidence.
